# AI Fleet Project Execution

## Mikor használd

Szabi átad egy projektet "csapatként vigyétek végig önállóan" instrukcióval. Triggerek:
- "Rátok bízom, csináljátok"
- "Folyamatosan adj feladatokat Zarának/Samunak/...", 
- "Nem akarok beleszólni, csak a heti riportot kérem"

A connectors.hu projekt minta (2026-05-06): MCP-as-a-Service Szabi voice msg-ből indítva.

## KANBAN-FIRST workflow (KÖTELEZŐ alapértelmezés, 2026-05-19 Szabi)

Minden projekt és részfeladat KÖTELEZŐEN kanban-kártyán keresztül menedzselt. Az ad-hoc inter-agent message-csak delegálás csak EMERGENCY-re (pl. broken-prod fix, time-critical pairing) -- minden más:

1. **Részfeladat → új kártya**: project mezővel (`connectors.hu`, `marveen-platform`, stb.), assignee mezővel (Samu/Zara/...), priority + status='planned' vagy 'in_progress'.
2. **Delegálás → inter-agent message + kanban referencia**: az üzenetben hivatkozz a kártya ID-ra (`Lásd kanban kártya XXXX, status=in_progress`). Az agent a kártyán is jelezzen vissza (`kanban_comments` táblába írjon kommentet).
3. **Sub-agent kötelezettsége**: az agent CLAUDE.md-jébe építve, hogy minden feladaton STATUS-OLJON (planned → in_progress → waiting/done). NEM csak inter-agent reply.
4. **Marveen 3-4 órás audit**: scheduled task (`~/.claude/scheduled-tasks/kanban-audit/`) ami 3 óránként:
   - Tisztítja a 7+ napos done-okat (archive)
   - Beakadt taskokat (>24h in_progress mozgás nélkül) számon kéri az assignee-nél inter-agent message-szel
   - Új delegálatlan/nélküli kártyákat jelzi Marveen-nek
5. **Heti riport**: Marveen összegzi a board-állapotot Telegramon Szabinak (cn00x002 kártya: minden péntek).
6. **Meta-task NEM kerül kanbanra**: ami folyamatos monitor / scheduled-task / mindenkori `/loop`-ciklus (pl. "GitHub PR monitor", "Reggeli napindító"), ne legyen kanban-kártya — mert a kanban-audit beakadtnak fogja jelölni. Ezeket `~/.claude/scheduled-tasks/<név>/` mappában tartsuk, vagy a `~/.claude/scheduled-tasks.json`-ben. (Samu 2026-05-19 08:01 feedback alapján: 258C8D8A kártya done-ba zárva mert a pr-figyeles scheduled-task-ja kezeli.)

Korábban a connectors.hu launch-nál működött ez (cn00m001-cn00s011 kártyák kanban-on át menedzselve). Utána ad-hoc lett a flow, ami zajos -- Szabi 2026-05-19 hangüzenetben visszahozta mint default.

## Eljárás

### 1. Indítás (~30 perc)

1. **Kanban parent kártya**: hozz létre egy nagy hierarchia-fa parent-kártyát a projektnek (id pl. `5d37b138`, title "X-projekt -- projektvezetés"). Description: scope, célok, szabályok.

2. **Sub-kártyák**: bontsd a projektet ágensonkénti sub-task listára. Konvenció: `<projekt>m###` (Marveen marketing-koordináció / DNS / heti riport), `<projekt>s###` (Samu dev), `<projekt>n###` (urgent crash-targets), stb. Minden sub-card: assignee, prioritás, leírás, sort_order.

3. **Inter-agent kickoff**: küldj részletes task-listát az ágenseknek inter-agent msg-vel. Külön Zara (üzleti/marketing) és Samu (technikai/fejlesztés). Brief mindkét oldalt, és figyelmezesd hogy NEM auto-pull (csak akkor dolgozik amikor pingelsz).

4. **Heti riport scheduled task**: hozz létre `~/.claude/scheduled-tasks/<projekt>-weekly-report.md` cron-os jelentés-küldővel. Péntek 17:00, kanban státusz + ágens-haladás.

5. **HOT memória**: mentsd el a projekt jelen állapotát (parent + sub kártya ID-k, scope, deadlines).

### 2. Folyamatos koordináció

- **Inter-agent msg = task delegálás**, **kanban = state tracking**. Egyik nélkül se megy.
- **Minden ágens-jelentésre**: 1) Kanban kártya frissítés (status, comment), 2) Inter-agent nyugta + következő task, 3) Telegram update Szabinak CSAK ha döntés kell tőle vagy fontos milestone.
- **Párhuzamos taskok**: ha egy ágensnek nincs blokkolt taskja, NE várj, adj neki proaktív "ha unatkozol" feladatot (versenytárs analízis, README docs, stb.).
- **Brief revíziók**: minden review-d konkrét legyen (1, 2, 3 pontos lista a változtatandóakra). Ne általános észrevétel ("nem tetszik"), hanem konkrét string cseréket adj.

### 3. Stack pivot kezelés

Szabi gyakran pivot-ol architektúrát egy közbejövő ötlet alapján (5 alkalom 1 nap alatt a connectors.hu projektnél). Pivot-okra reagálva:

1. **Kanban update**: parent + érintett sub kártyák kommentbe pivot-info (mire váltottunk és miért).
2. **Inter-agent msg**: a változott scope-pal lerendelni az ágenseket. Pivot UTÁN csinált munkát NEM eldobni, hanem portolni / újrahasznosítani amennyit lehet (a kód 90%-a általában megmarad).
3. **Decision log memória** (cold tier): rögzítsd a pivot-okat időbélyegekkel ("11:25 Szabi stack pivot CF->Supabase only").
4. **Időbecslés**: ne ember-órákban becslés (Szabi explicit korrigált). AI tempó ~5-10x gyorsabb mint humán fejlesztő. 8 órás meló = 1 óra AI-flotta-tempó.

### 4. Deploy-blokker kezelés

A flotta nem tud Cloudflare/Netlify/Stripe-fiókot regisztrálni, mert:
- computer-use Chrome tier=read (kattintás blokkolva)
- Nincs claude-in-chrome MCP általában
- Account-creation API-n keresztül nem support-olt

Ezért a deploy-blokkerek mindig Szabira várnak. Mitigation:
- **Bundle a kéréseket egy Telegram-msg-be** (CF token + Account ID + Netlify connect + DNS CNAME = egy üzenet, Szabi 5-10 perc alatt csinálja).
- **Stack-egyszerűsítést mérlegelj**: ha az architektúra Supabase-only-vá redukálható (Edge Functions = Worker proxy egyenérték MVP-hez), drop-old a Cloudflare-t teljesen. Egy vendor = egy deploy-blokker.

- **PR-merge engedély: judgment-based, NEM minden PR-nél kötelező** (frissítve 2026-05-11): Szabi explicit feedback "ha részedről OK, csináljátok a merge-et" — proaktív merge kell ha:
  - Samu/Iris/Zara végzett egy PR-t, build OK
  - A diff diff egyértelmű, alacsony kockázat (UI bugfix, copy update, content tweak)
  - Nincs irreverzibilis hatás (nem migration, nem prod-data, nem money-flow)
  → MERGELJEM AUTOMATIKUSAN, csak update-üzenet Szabinak hogy merged.
  
  KÉRDEZZEK Szabit MIELŐTT mergelnél, ha:
  - DB migration (irreverzibilis schema-change)
  - Production-affecting (money-flow, user-credits, encrypted credentials)
  - Architecture-pivot (új feature scope, ár-változtatás, branding)
  - Stack-PR (több függő PR egymáson, sorrend-érzékeny)
  - Bizonytalanság a PR diff-jében
  
  Reason: a túl-cautious "minden PR-nél engedélyt kérek" lassítja a flotta-throughput-ot. Szabi a high-confidence routine UI/copy-fixeket gyorsabban akarja átengedni.

Az ELŐZŐ szabály ("soha ne chain-eld") MARAD a `&&` bash-parancsra (mert egyetlen tool-call-ban túl gyors, Szabi nem tud közbevágni) — de a tool-call-szintjéig elkülönített `gh pr create` + várás + `gh pr merge` LOW-risk PR-nél engedélyezett.

- **PR merge után-push race: post-merge branch commit ELVÉSZ** (2026-05-11 Zara PR #68): ha egy ágens PR-t mergelünk, és a child-agent UTÁNA pusholja az updated commit-okat ugyanarra a (most már zárt) branch-re, azok a változások **nem kerülnek main-re** (a PR már merged). - **Cascade conflict párhuzamos PR-eknél azonos fájl-területen** (2026-05-18 Samu security 4 PR connectors-api): amikor egy sub-agent egyszerre több feature-PR-t nyit ugyanazon main-szinten (mind külön branch, ugyanahhoz a base-commitre alapozva), és minden PR módosítja ugyanazt a fájlt (pl. `index.ts` router), a sorrendi merge minden iterációnál újra conflict-ot dob a maradékra. Konkrétan: PR #58 #59 #60 #61 mind a router-be új if-blokkot rakott; #58 merge után #59-#61 conflict, #59 merge után #60-#61 conflict, #60 merge után #61 conflict — 3 cascade-rebase kör. **Megoldás A** (preferált): amikor delegálsz egy multi-PR feature-csomagot egy sub-agent-nek, **kérd hogy egy ÚJ branch-en chain-elve build-elje** (PR1 base = main, PR2 base = PR1, PR3 base = PR2, stb.) — így a merge sorrendje deterministic és cascade-rebase nincs. **Megoldás B** (fallback ha már külön branch-ek): sub-agent-tel intézd a `git rebase origin/main && git push -f` műveletet PR-enként sorban, NE próbáld te magad rebase-elni a sub-agent munkáját. Plus: 4+ PR-es feature-csomagnál érdemes ELŐRE jelezni a sub-agentnek: „**branch-strategy**: vagy chain-elt branch-ek (PR1 base = main, PR2 base = PR1, stb.), vagy egy nagy bundled PR". Egy 4-PR cascade kb. 10 perc extra rebase-loop, de chain-elt branch-esetén 0. 5+ PR-nél a bundled-PR is OK, ha review-méret kezelhető (≤1500 sor).

## Buktató — Chart.js világos témán hardcoded színek

**Probléma** (2026-05-18 admin PR #101): a chart-config tick + grid colors hardcoded `rgba(255,255,255,0.3-0.5)` voltak — sötét témára tervezve. De a user a világos témán nyitotta meg, és a label-ek láthatatlanok voltak.

**Megoldás**: minden Chart.js `ticks.color`, `grid.color`, `tooltip.bodyColor` érték legyen CSS-var-driven, runtime-ban `getComputedStyle(document.documentElement).getPropertyValue('--ink-2').trim()` kifejezésével. Tipikus változó-mapping: `--ink-2` → tick color, `--border` → grid color, `--accent` → primary fill/stroke. Fallback dark-themed value, de a runtime-érték nyer mindig.

Snippet:
```ts
const css = (name: string, fb: string) => getComputedStyle(document.documentElement).getPropertyValue(name).trim() || fb;
const tickColor = css('--ink-2', 'rgba(0,0,0,0.55)');
const gridColor = css('--border', 'rgba(0,0,0,0.08)');
```

**A delegáció-promptban Zarának (frontend agent)**: amikor chart-renderelést kérsz, EXPLICIT írd, hogy "tick + grid colors CSS-var-driven (`--ink-2`, `--border`), NEM hardcoded".
